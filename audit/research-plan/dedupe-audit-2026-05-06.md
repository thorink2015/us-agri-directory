## Caveats on the parser

- Markdown tables in batches 4a/4b/5 mix multiple sub-tables
  (operator listings, summary stats, regulatory references).
  Without per-section context, the parser captures some non-operator
  rows. Manual review by Eugen is the final arbiter.
- The fuzzy-match strips `LLC`, `Inc`, `Corp`, etc. but does not
  catch reorderings ("ABC Drone Services" vs. "Drone Services
  ABC") or material renames. Misses below this threshold are
  possible but should be rare.
- Markdown-table parsing relies on the first column being the
  operator name. Two batches use this convention; one (mountain-
  west-v2) places state-name section markers as table rows, which
  the noise filter removes.

## Snapshot drift on `existing-operators-name-list.txt`

The 386-row list is missing 6 operators that exist in
`operators.ts` (the names listed at the top of "Top-level findings").
The list is clearly a stale snapshot rather than a generated
artifact. If it's used as the dedupe source for any future research
prompt, it will under-report, causing the prompt to "rediscover"
operators that already exist.

**Recommend:** regenerate the list from `operators.ts` whenever a
research batch is about to run. A one-liner:

```bash
grep -E "^\s+name:" src/data/operators.ts | \
  sed -E "s/.*name: '([^']+)'.*/\1/" | sort > \
  audit/research-plan/existing-operators-name-list.txt
```

(Add city/state suffix from a richer query if the prompt requires
disambiguation.)
