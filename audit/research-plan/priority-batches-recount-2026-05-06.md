## Method

For each row in `priority-batches.md`:
1. Parse `City`, `State`, and recorded `Existing ops`.
2. Slugify `State` (e.g. `New Mexico → new-mexico`).
3. Count operators in `src/data/operators.ts` where
   `op.city == City` (case-insensitive) **and** the state slug
   appears in `op.counties`.
4. Apply the same coverage thresholds priority-batches uses
   (`0 → gap`, `1–2 → light`, `≥3 → covered_well`).
5. Flag any row where the recount or status differs.

State-slug sanity: every distinct state name in priority-batches
(34 states) maps to a slug that exists in operators.ts. No
unmapped states.

## Why the snapshot held up

The 393 operators in `operators.ts` accumulated across many PRs in
April and early May. Most of those imports targeted cities that
were already at `light` or `covered_well` (i.e. existing
directory cities). The priority-batches list deliberately
selected `gap` and `light` cities — by definition the cities the
April–May import waves were _not_ targeting. Hence the small
drift.

## Caveats

- Match is on `op.city == City` AND state slug ∈ `op.counties`.
  `counties` is mislabeled in the schema (it actually holds state
  slugs). An operator HQ'd in Lubbock TX with no Texas tag in
  `counties` would not be counted — Eugen should sanity-check
  that all operators are tagged with at least their home state.
- Cross-state city-name collisions (e.g. Springfield, IL vs.
  Springfield, MO) are correctly disambiguated by the state slug
  filter.
- `master-cities-ranked.csv` (860 rows) was not used for this
  recount — only the 200-row priority-batches.md slice. Recount
  for the rest of master-cities is unnecessary unless you want to
  expand priority-batches.
