# Import script extension spec — 2026-05-06

> Technical spec for extending `scripts/import-operators.ts` to
> handle the new `audit/research-plan/batch-*-results.md`
> deliverable format. Plus the citation-persistence and geocoding
> open decisions, framed so each can be implemented independently.
>
> Spec only. No code changes. Generated by Claude Code on session
> `claude/review-codebase-docs-sJEBA`. Intended to be the
> implementation reference for the May-16 freeze-end merge session.

## Current script (baseline)

`scripts/import-operators.ts` (541 lines) does the following today:

1. **Reads 7 hard-coded `_research/operators-batch-*.md` files** —
   list literal at the top, no auto-discovery.
2. **Auto-detects format** per file: CSV in a fenced ```csv block,
   or a markdown pipe-table. Returns `RawRow[]` from either.
3. **Maps headers via fuzzy regex** in `mapToOperator()` —
   `getCol(row, /company.*name/i, /^company$/i, /^name$/i)` style.
   Tolerant of header variation across batch files.
4. **Builds a `MappedOperator`** with optional fields and
   exclusion of empty values via spread. Adds UTM tags to the
   website URL.
5. **Filters by category** — keeps `operator`, `dealer`, `multi`,
   `O`, `D`. Excludes `training`, `manufacturer`, `government`,
   `university`, `extension`, `research`, `franchisor`,
   `T`, `M`. Matches the dedupe-audit observations (T1.4).
6. **Dedupes against existing** `src/data/operators.ts` (loads
   names case-insensitively).
7. **Internal cross-batch dedupe** — first occurrence wins.
8. **Writes `scripts/import-dry-run.json`** — does **not** modify
   `operators.ts`.

The script is therefore a dry-run reporter. The actual write to
`operators.ts` is currently a manual paste step (or a follow-up
script that's not in the repo).

## What needs to change

### 1. Add `audit/research-plan/batch-*-results.md` as a source path

The new research-plan deliverables don't exist yet (Batch 1 stalled
in the prior session — see `_memory/known-issues.md` 2026-05-06
entry). When Batch 1 actually runs, the file will land at
`audit/research-plan/batch-01-results.md`.

**Change:** swap the `BATCH_FILES` literal for a glob:

```ts
import { globSync } from 'glob';

const BATCH_FILES = [
  ...globSync('_research/operators-batch-*.md'),
  ...globSync('audit/research-plan/batch-*-results.md'),
];
```

Or, if you want to avoid the `glob` dep, use `fs.readdirSync` over
both directories. Order doesn't matter (the first-occurrence
dedupe runs after the merge).

**Note on `_research/operators-batch-4b-mountain-west.md` (no -v2)**
— the dedupe audit (T1.4) showed this file parses to 0 rows, so
the existing import never picked it up. If `-v2` superseded it,
keep `-v2` and either delete the old file or add it to a deny
list. If both should be processed, the format mismatch in the
non-v2 file needs investigation first.

### 2. Standardize the new-batch schema in the prompt

Today the parser tolerates variation because the markdown files
were hand-authored. The new prompts should **emit a canonical
schema** so the parser doesn't have to keep growing fuzzy
patterns. Recommend the prompt produce a markdown pipe-table with
exactly these columns, in this order:

```
| Company | Category | City | State | States Served | Drones | Services | Crops | Phone | Email | Website | Facebook | Instagram | LinkedIn | YouTube | Founded | Part 107 | Part 137 | Description | Source URL |
```

This matches the existing `getCol` regexes (no fuzzy matching
needed) and pre-aligns with the existing parser's column resolver.

If the prompt uses CSV instead of pipe-table, fence with ```csv
(triple backtick + `csv`) so `parseCsvBlock()` picks it up.

### 3. Dedupe across all three sources

The current dedupe stack:

```
research-files → category filter → dedup vs operators.ts → dedup cross-batch → final
```

For the multi-source case (legacy `_research/` + new
`audit/research-plan/`), the dedupe order should be:

```
all-sources merged → category filter → dedup vs operators.ts → dedup cross-source → final
```

The existing logic already does this — the only change is that
"all-sources" includes both globs. **Edge case to guard:** if a
research file in `_research/` and a new file in
`audit/research-plan/` describe the same operator with slightly
different name spellings ("XYZ Drone Services LLC" vs "XYZ
Drone"), the case-insensitive exact-match dedupe will treat them
as different. Recommend adding a normalized-name comparator as a
second pass:

```ts
const normName = (s: string) =>
  s.toLowerCase()
   .replace(/\b(llc|inc|corp|co|company|ltd)\.?\b/g, '')
   .replace(/[^a-z0-9 ]/g, '')
   .replace(/\s+/g, ' ')
   .trim();
```

Apply during both `loadExistingNames()` and the cross-source
dedup. Surface near-matches to the dry-run JSON for human review
(don't auto-merge).

### 4. Citation persistence (open decision — pick one)

The CSV/table includes a `Source URL` column today. The current
script reads it via `getCol(row, /short.?description|^description$/i)`
for the description but ignores `Source URL` entirely. Three
options for the new import:

#### Option A: Add `sourceUrl?: string` field on `Operator`

Schema change, but cleanest. Add to `src/data/types.ts`:

```ts
export interface Operator {
  // ... existing
  sourceUrl?: string;
}
```

And to the operator profile template, render in the footer:
"Listed sources: [link]". No SEO impact (one extra outbound link
per operator). No cost. Best for E-E-A-T transparency.

#### Option B: Append to `description`

```ts
const description = clean(getCol(row, /short.?description/i)) +
  (sourceUrl ? `\n\nSources: ${sourceUrl}` : '');
```

Zero schema change. But mixes citation noise into the rendered
description body. Discouraged.

#### Option C: Markdown-only audit trail

Keep the `Source URL` in a separate file
`audit/research-plan/operator-citations.md` (one row per
imported operator) and never store on `Operator`. The directory
stays clean; citations remain auditable.

**Recommendation: A.** Schema change is one line; rendering is
one extra `<a>` per operator profile; the source link directly
strengthens E-E-A-T (which `_memory/standing-rules.md` flags as
a hard requirement).

### 5. Geocoding (open decision — pick one)

Operators in `operators.ts` have optional `lat: number`, `lng:
number` fields. The current import script doesn't set them; new
operators land without coordinates. Three options:

#### Option A: Wire Nominatim (OSM) into the import

```ts
async function geocode(city: string, stateSlug: string) {
  const q = `${city}, ${stateSlug.replace(/-/g, ' ')}, USA`;
  const res = await fetch(
    `https://nominatim.openstreetmap.org/search?format=json&q=${encodeURIComponent(q)}&limit=1`,
    { headers: { 'User-Agent': 'agdronedirectory.com import' } }
  );
  const [hit] = await res.json();
  return hit ? { lat: parseFloat(hit.lat), lng: parseFloat(hit.lon) } : null;
}
```

- Free, no API key.
- Hard rate limit: 1 request/sec. With ~200 new operators per
  research wave, that's ~3.5 minutes per import — acceptable.
- TOS requires User-Agent and respect for the rate limit.
- Cache results in `scripts/geocode-cache.json` keyed by
  `${city}|${stateSlug}` so re-runs skip already-resolved cities.

#### Option B: Use a static city-coordinates table

Pre-build `src/data/us-cities-coords.ts` from a Census Bureau
gazetteer file (one-time, ~30k US cities). Lookup is O(1), no
network. Tradeoff: city names not in the gazetteer (small towns,
ambiguous spellings) fall through to undefined.

#### Option C: Leave `lat`/`lng` blank for new operators

Map page already handles missing coordinates by skipping the
marker. State pages don't depend on per-operator coordinates.
Lowest risk; loses map-marker coverage for new operators.

**Recommendation: A with B fallback.** Try Nominatim first; if it
fails or returns nothing, fall back to the static gazetteer.
Cache aggressively. The map-marker coverage is part of the
directory's value prop.

### 6. Auto-write to `operators.ts` (optional)

Today the script writes JSON; humans paste. For unattended runs,
add a `--write` flag that:

1. Loads `src/data/operators.ts`
2. Locates the closing `];` of the `operators` array
3. Inserts `JSON.stringify(final, null, 2)` (without the outer
   brackets) immediately before the close
4. Runs Prettier on the file

Risk: if the AST of `operators.ts` ever differs from a flat
array literal, the regex insertion breaks. Recommend keeping
`--write` disabled by default and printing the exact insertion
point for manual confirmation on the first run.

## Summary of decisions to make before May 16

| Decision | Recommended | Rationale |
|---|---|---|
| Source-path glob | Add both globs | Backward-compatible; one-line change |
| Canonical batch schema | Pipe-table with the 20 columns above | Aligns with existing `getCol` patterns |
| Dedupe | Add `normName` second pass | Catches LLC/Inc spelling variants |
| Citation field | A: `sourceUrl?: string` | E-E-A-T win; one schema line |
| Geocoding | A with B fallback | Map coverage matters; cache mitigates rate limit |
| Auto-write | Defer | Manual paste is low-friction at the current scale |
